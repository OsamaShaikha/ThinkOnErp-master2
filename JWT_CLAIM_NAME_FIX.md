# JWT Claim Name Mismatch Fix

## Error
```
System.UnauthorizedAccessException: User ID not found in token
at ThinkOnErp.API.Controllers.SavedSearchesController.GetCurrentUserId()
```

## Root Cause

**Mismatch between JWT token generation and claim extraction:**

### What JwtTokenService Generates:
```csharp
var claims = new[]
{
    new Claim("userId", user.RowId.ToString()),      // ← lowercase "userId"
    new Claim("userName", user.UserName),
    new Claim("role", user.Role?.ToString() ?? "0"),
    new Claim("branchId", user.BranchId?.ToString() ?? "0"),
    new Claim("isAdmin", user.IsAdmin.ToString().ToLower())
};
```

### What SavedSearchesController Was Looking For:
```csharp
var userIdClaim = User.FindFirst(ClaimTypes.NameIdentifier)?.Value  // ← Wrong claim type
    ?? User.FindFirst("UserId")?.Value;                              // ← Wrong case (capital U)
```

**Result:** The controller couldn't find the user ID because it was looking for the wrong claim names!

---

## Solution Applied

Changed `SavedSearchesController.GetCurrentUserId()` to look for the correct claim name:

### Before (Incorrect):
```csharp
private Int64 GetCurrentUserId()
{
    var userIdClaim = User.FindFirst(ClaimTypes.NameIdentifier)?.Value 
        ?? User.FindFirst("UserId")?.Value;  // Wrong case!
    
    if (string.IsNullOrEmpty(userIdClaim) || !Int64.TryParse(userIdClaim, out var userId))
    {
        throw new UnauthorizedAccessException("User ID not found in token");
    }

    return userId;
}
```

### After (Correct):
```csharp
private Int64 GetCurrentUserId()
{
    var userIdClaim = User.FindFirst("userId")?.Value  // ← Correct: lowercase "userId"
        ?? User.FindFirst(ClaimTypes.NameIdentifier)?.Value;  // Fallback
    
    if (string.IsNullOrEmpty(userIdClaim) || !Int64.TryParse(userIdClaim, out var userId))
    {
        throw new UnauthorizedAccessException("User ID not found in token");
    }

    return userId;
}
```

## How to Extract Claims in Controllers

### ✅ Correct Way:

```csharp
// Get user ID
var userId = User.FindFirst("userId")?.Value;

// Get username
var userName = User.FindFirst("userName")?.Value;

// Get role
var role = User.FindFirst("role")?.Value;

// Get branch ID
var branchId = User.FindFirst("branchId")?.Value;

// Check if admin
var isAdmin = User.FindFirst("isAdmin")?.Value == "true";

// Check if super admin
var isSuperAdmin = User.FindFirst("isSuperAdmin")?.Value == "true";
```

### ❌ Wrong Way:

```csharp
// DON'T use ClaimTypes.NameIdentifier - it's not in our tokens
var userId = User.FindFirst(ClaimTypes.NameIdentifier)?.Value;

// DON'T use wrong case
var userId = User.FindFirst("UserId")?.Value;  // Wrong! Should be "userId"
```

## Standard Claim Types vs Custom Claims

### Standard ClaimTypes (NOT used in our system):
- `ClaimTypes.NameIdentifier` = "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier"
- `ClaimTypes.Name` = "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
- `ClaimTypes.Role` = "http://schemas.microsoft.com/ws/2008/06/identity/claims/role"

### Our Custom Claims (USED in our system):
- `"userId"` - Simple string, lowercase
- `"userName"` - Simple string, camelCase
- `"role"` - Simple string, lowercase
- `"branchId"` - Simple string, camelCase
- `"isAdmin"` - Simple string, camelCase

**Why custom claims?**
- Shorter claim names = smaller JWT tokens
- More control over claim structure
- Easier to read and debug

## Prevention

To avoid this issue in the future:

### 1. Create a Helper Class:

```csharp
public static class ClaimsPrincipalExtensions
{
    public static long GetUserId(this ClaimsPrincipal principal)
    {
        var userIdClaim = principal.FindFirst("userId")?.Value;
        if (string.IsNullOrEmpty(userIdClaim) || !long.TryParse(userIdClaim, out var userId))
        {
            throw new UnauthorizedAccessException("User ID not found in token");
        }
        return userId;
    }

    public static string GetUserName(this ClaimsPrincipal principal)
    {
        return principal.FindFirst("userName")?.Value 
            ?? throw new UnauthorizedAccessException("User name not found in token");
    }

    public static long? GetBranchId(this ClaimsPrincipal principal)
    {
        var branchIdClaim = principal.FindFirst("branchId")?.Value;
        return long.TryParse(branchIdClaim, out var branchId) ? branchId : null;
    }

    public static bool IsAdmin(this ClaimsPrincipal principal)
    {
        return principal.FindFirst("isAdmin")?.Value == "true";
    }

    public static bool IsSuperAdmin(this ClaimsPrincipal principal)
    {
        return principal.FindFirst("isSuperAdmin")?.Value == "true";
    }
}
```

### 2. Use in Controllers:

```csharp
public class SavedSearchesController : ControllerBase
{
    [HttpGet]
    public async Task<ActionResult> GetSavedSearches()
    {
        var userId = User.GetUserId();  // ← Extension method
        var userName = User.GetUserName();
        var isAdmin = User.IsAdmin();
        
        // ... rest of code
    }
}
```

### 3. Benefits:
- ✅ Centralized claim extraction logic
- ✅ Consistent across all controllers
- ✅ Easy to update if claim names change
- ✅ Better error messages
- ✅ Type-safe

## Summary

**Problem:** SavedSearchesController was looking for `ClaimTypes.NameIdentifier` or `"UserId"`, but the JWT token contains `"userId"` (lowercase).

**Solution:** Changed the controller to look for `"userId"` first, matching what `JwtTokenService` generates.

**Impact:** SavedSearches API now works correctly with JWT authentication!

**Recommendation:** Create extension methods for claim extraction to prevent this issue in other controllers.

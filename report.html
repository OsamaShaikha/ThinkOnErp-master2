Permission Model

Each screen supports 4 granular permission flags, managed at both the role and individual user level.

Permission Types

Every screen can have these 4 independent permissions assigned:

V  Can View

Allows viewing and reading screen data. The most basic access level.

I  Can Insert

Allows creating new records. Typically paired with View permission.

U  Can Update

Allows editing existing records. Requires View permission to be useful.

D  Can Delete

Allows soft-deleting records. Highest privilege level per screen.

Permission Hierarchy

Permissions flow from role to user, with user-level overrides taking precedence:

 Role-Screen Permissions

Assigned to a role via SYS_ROLE_SCREEN_PERMISSION. All users with that role inherit these permissions.

 User-Screen Overrides

Per-user overrides via SYS_USER_SCREEN_PERMISSION. Takes precedence over role permissions. Includes notes and audit trail.

 Company-System Access

Companies subscribe to systems via SYS_COMPANY_SYSTEM. Only subscribed systems' screens are accessible.

Screen & Permission API Endpoints

Manage screens and permissions programmatically via the REST API.

GET /api/permissions/screens — Get all screens for a system
GET /api/permissions/screens/{id} — Get screen by ID
POST /api/permissions/screens — Create a new screen
PUT /api/permissions/screens/{id} — Update a screen
DEL /api/permissions/screens/{id} — Delete a screen (soft delete)
GET /api/permissions/roles/{roleId}/screens — Get role screen permissions
POST /api/permissions/roles/{roleId}/screens — Assign screen permissions to role
GET /api/permissions/users/{userId}/screens — Get user screen permissions
POST /api/permissions/users/{userId}/screens — Assign screen permissions to user
GET /api/permissions/users/{userId}/effective — Get effective (merged role + user) permissions

Database Schema

Underlying Oracle tables that store screen and permission data.

Table Purpose Key Columns
SYS_SYSTEM ERP system modules SYSTEM_CODE, SYSTEM_NAME, SYSTEM_NAME_E, ICON, IS_ACTIVE
SYS_SCREEN Individual screens within systems SYSTEM_ID, PARENT_SCREEN_ID, SCREEN_CODE, SCREEN_NAME, ROUTE, ICON
SYS_COMPANY_SYSTEM Company ↔ System subscriptions COMPANY_ID, SYSTEM_ID, IS_ALLOWED, GRANTED_BY
SYS_ROLE_SCREEN_PERMISSION Role-level screen permissions ROLE_ID, SCREEN_ID, CAN_VIEW, CAN_INSERT, CAN_UPDATE, CAN_DELETE
SYS_USER_SCREEN_PERMISSION User-level permission overrides USER_ID, SCREEN_ID, CAN_VIEW, CAN_INSERT, CAN_UPDATE, CAN_DELETE, NOTES
SYS_USER_ROLE User ↔ Role assignments USER_ID, ROLE_ID, ASSIGNED_BY, ASSIGNED_DATE
